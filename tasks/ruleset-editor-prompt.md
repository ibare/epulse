# ePulse 룰 파라미터 튜닝 화면 구현 프롬프트

당신은 React 기반 정적 웹앱을 구현하는 시니어 프론트엔드 엔지니어다.  
이 작업은 ePulse의 **룰 파라미터 튜닝 화면(Settings / Rule Tuning Panel)** 을 구현하는 것이다.

이 프로젝트는 실전 투자 도구가 아니라 **경제 인과관계 학습 도구**다.  
정확한 예측보다 **설명력, 직관성, 시각적 명확성, 아름다운 UI**가 우선이다.

---

## 1. 목표

메인 거시 화면에서 사용자가 `설정` 버튼을 누르면,  
현재 앱이 사용하는 **룰셋의 구조를 시각적으로 탐색**하고,  
각 룰의 **조정 가능한 숫자(weight, threshold 등)** 를 수정할 수 있는 설정 화면 또는 설정 패널을 제공하라.

이 기능의 목적은 다음과 같다.

- 룰 구조를 눈에 보이게 한다
- 숫자 하드코딩 문제를 해결한다
- 전문가/고급 사용자가 민감도를 튜닝할 수 있게 한다
- 코드 수정 없이 런타임에서 모델을 조정할 수 있게 한다

중요:
- **룰의 구조는 수정 불가**
- **숫자 파라미터만 수정 가능**
- **수정값은 localStorage에 저장**
- **룰셋 버전이 다르면 저장본은 로드하지 않음**

---

## 2. 핵심 철학

이 화면은 “코드 설정 화면”이 아니라  
**시각적으로 잘 정리된 모델 파라미터 스튜디오**여야 한다.

즉 다음을 동시에 만족해야 한다.

- 전문적인 느낌
- 지나치게 개발자 도구 같지 않을 것
- 초보자가 봐도 공포스럽지 않을 것
- 숫자를 바꾸는 행위가 결과 모델에 영향을 준다는 감각을 줄 것

---

## 3. 사용자 시나리오

### 기본 흐름
1. 메인 화면 우측 상단의 `설정` 버튼 클릭
2. 설정 패널 또는 별도 설정 화면 진입
3. `룰 튜닝` 섹션 선택
4. 룰 목록 탐색
5. 특정 룰의 가중치/임계값 수정
6. 수정 즉시 런타임에 반영
7. 수정값 localStorage 저장
8. 앱 재진입 시 같은 버전이면 로드
9. 앱 버전이 바뀌었으면 이전 저장값은 무시

---

## 4. 수정 가능 범위

사용자가 수정할 수 있는 것은 **숫자 파라미터만**이다.

### 수정 가능
- rule weight
- activation threshold
- 상태 라벨 경계값 (선택)
- global decay factor
- 시차 강도 계수 (선택)

### 수정 불가
- source node
- target node
- 방향 (+ / -)
- rule 삭제
- rule 추가
- exception 구조 변경
- layer 변경
- region 변경

즉, **topology는 고정하고 parameter만 조정 가능**해야 한다.

---

## 5. 룰셋 버전 처리 (필수)

룰셋은 향후 변경될 수 있으므로 **versioned persistence**를 구현하라.

### 요구사항
- 앱 내부에 `RULESET_VERSION` 상수를 둔다
- localStorage에 저장할 때 version을 함께 저장한다
- 앱 시작 시 localStorage의 saved version과 현재 RULESET_VERSION을 비교한다
- 버전이 같을 때만 저장된 튜닝값을 로드한다
- 버전이 다르면 저장본을 무시하고 기본값을 사용한다

### 동작 방식
- mismatch 시 에러를 내지 말 것
- 조용히 무시해도 되지만, 설정 화면에서 작은 안내 배지를 보여주는 것은 허용
- 예:
  - `이전 버전 설정은 현재 룰셋과 호환되지 않아 적용되지 않았습니다`

### 저장 구조 예시
```ts
{
  version: "v1.3.0",
  updatedAt: 1712345678901,
  rules: {
    r08: { weight: 0.9 },
    r09: { weight: 0.25 }
  },
  globals: {
    activationThreshold: 4,
    decayFactor: 0.7
  }
}
```

## 6. 저장 정책

### localStorage 키
하나의 명확한 키를 사용하라.

예:
- `epulse.ruleset.tuning`

### 저장 시점
- 사용자가 숫자를 수정할 때 debounce 저장
또는
- 수정 즉시 저장

둘 다 가능하지만 UX상 버벅임이 없도록 구현

### 초기화
- `기본값 복원` 버튼 제공
- 클릭 시 현재 버전 기준 기본값으로 복원
- localStorage도 함께 초기화

---

## 7. 화면 구조

설정 화면 또는 드로어/모달은 아래 구조를 따른다.

### 상단
- 제목: `설정`
- 서브탭:
  - 일반
  - 룰 튜닝
  - 정보
- 현재 룰셋 버전 표시
- 기본값 복원 버튼

### 룰 튜닝 화면 본문
좌우 2단 또는 3단 레이아웃 권장

#### 왼쪽: 필터/탐색
- 검색 입력
- 지역 필터: KR / US / GL / ALL
- 레이어 필터: cause / transmission / market / ALL
- 상태 필터: 수정됨 / 기본값 / 전체

#### 중앙: 룰 목록
각 룰을 카드 또는 행 단위로 표시

표시 정보:
- 룰 ID
- source → target
- 방향
- 설명
- 현재 weight
- 기본 weight
- 시차
- 예외 조건 요약 (있을 경우)
- 수정 여부 배지

#### 오른쪽: 상세 편집 패널
룰 선택 시 상세 정보 표시

표시 요소:
- 룰 이름 / ID
- source / target
- 방향
- 설명
- weight slider + numeric input
- threshold input (존재 시)
- 관련 노드 미리보기
- 이 룰이 영향을 주는 의미 설명

## 8. UI 디자인 원칙

이 화면은 매우 중요하다.  
설정 화면이라고 해서 투박하거나 개발자 도구처럼 보이면 안 된다.

### 스타일 방향
- 메인 앱과 동일한 다크 테마
- 반투명 카드/패널
- 얇은 보더와 소프트 글로우
- 정돈된 타이포그래피
- 밀도는 높되 답답하지 않게

### 시각적 인상
- “관리자 패널”보다는 “정교한 모델 콘솔”
- “JSON 편집기” 같은 느낌 금지
- 파라미터를 만지는 행위가 고급스럽고 신뢰감 있게 느껴져야 한다

### 색상 원칙
- 기본 상태: 중립 회색/슬레이트
- 수정된 값: 청록/파랑 계열 강조
- 위험하거나 과도한 값: 주황 계열 경고
- 기본값 복원 / 버전 mismatch 등은 은은한 주황/노랑 배지

### 카드 표현
룰 카드는 다음 속성을 가져야 한다.
- ID 배지
- 설명성 높은 제목
- source/target 관계 시각화
- 현재값 vs 기본값 비교
- 미세한 hover 반응
- 선택 시 주변 glow 강조

---

## 9. 룰 목록의 표현 방식

각 룰은 사람이 읽기 쉬운 문장처럼 보여야 한다.

### 예시 표현
- `한국 물가 → 한국 시장금리`
- `미국 기준금리 → 달러 강세 지수`
- `시장환율 기대 → 외국인 자금`

### 방향 표현
- 같은 방향: `+`
- 반대 방향: `-`

이 기호만 보여주지 말고 텍스트도 보조로 넣어라.
예:
- `같은 방향`
- `반대 방향`

### 시차 표현
칩/배지 형태로 표시
- 즉시
- 단기
- 중기

---

## 10. 편집 가능한 컨트롤

### weight
- slider + number input 동시 제공
- 정밀 조정 가능해야 함
- 범위 제한 예:
  - `0 ~ 1.5`
- 소수점 2자리 허용

### threshold
존재하는 경우에만 표시
- number input
- 범위 제한 예:
  - `0 ~ 20`

### decay factor / global params
룰별 리스트와 별도로 상단 또는 하단에 `전역 파라미터` 섹션 제공
예:
- activationThreshold
- decayFactor

이들도 동일하게 slider + numeric input 제공 가능

---

## 11. 수정 상태 표시

룰이 기본값에서 변경되었는지 바로 보여야 한다.

### 요구사항
- 수정된 룰에는 `Modified` 배지 표시
- 기본값과 현재값을 같이 표시
- 예:
  - `현재 0.90 / 기본 0.80`
- 기본값으로 되돌리는 작은 버튼 제공

### 전역 상태
상단에 요약 표시 가능
- `총 6개 룰 수정됨`
- `전역 파라미터 2개 수정됨`

---

## 12. 실시간 반영

사용자가 값을 바꾸면 **런타임 계산에 즉시 반영**되어야 한다.

### 요구사항
- 슬라이더 이동 시 그래프/상태 계산이 바로 반영
- 메인 화면 또는 상세 화면이 열려 있다면 그 상태에도 반영 가능
- 최소한 현재 세션 상태에는 즉시 반영될 것

### 구현 방식
- tuned ruleset을 앱 전역 상태에 merge
- 엔진은 항상 “기본 룰셋 + 사용자 튜닝값”으로 평가

---

## 13. 데이터 구조 설계

룰셋 구조와 튜닝 상태를 분리하라.

### 기본 룰 정의
고정 구조:
- id
- source
- target
- direction
- label
- description
- lag
- defaultWeight
- defaultThreshold
- exception
- editableParams

### 사용자 튜닝값
가변 구조:
- weight
- threshold
- global param overrides

### 최종 런타임 계산
- `resolvedRuleset = merge(defaultRuleset, tuningOverrides)`

## 14. 버전 불일치 처리 UX

버전이 맞지 않으면 localStorage를 로드하지 않는다.  
이때 UX는 조용하지만 이해 가능해야 한다.

### 권장 UX
설정 화면 상단에 작은 정보 배지 또는 인포 카드:
- `이전 저장 설정은 현재 룰셋 버전과 달라 적용되지 않았습니다.`
- `현재 버전: v1.3.0`

### 금지
- intrusive modal
- 경고창 남발
- 앱 진입 방해

## 15. 검색 및 필터 UX

룰이 많아질 수 있으므로 탐색성이 매우 중요하다.

### 검색
다음 기준으로 검색 가능:
- rule id
- source label
- target label
- description 일부

### 필터
- region
- layer
- modified only

### 정렬
- id 순
- source 기준
- modified 우선
중 하나 선택 가능하면 좋지만 필수는 아님

## 16. 시각적 관계 보조

설정 화면에서도 룰이 단지 표가 아니라 **관계**로 느껴져야 한다.

### 권장 요소
- selected rule의 source와 target를 작은 노드 미리보기 형태로 표현
- 간단한 arrow diagram 표시
- 예:
  - `[한국 물가] --(+0.60)--> [한국 시장금리]`

이 미리보기는 매우 중요하다.  
숫자만 편집하면 사용자는 맥락을 잃는다.

## 21. 최종 결과물의 인상

사용자는 이 화면을 보고 다음처럼 느껴야 한다.

- “아, 이 앱은 룰이 어떻게 작동하는지 투명하게 보여주는구나”
- “숫자를 바꾸면 모델 반응이 달라지는구나”
- “복잡하지만 잘 정리되어 있고 만져보고 싶다”
- “개발자 도구가 아니라 아름답게 설계된 모델 튜닝 화면이다”

## 22. 설계 원칙

선택이 필요할 때는 아래를 우선하라.

1. 구조 고정, 숫자만 수정 가능
2. 설명력 > 구현 편의
3. 아름다움 > 투박한 기능 나열
4. 탐색 가능성 > 테이블 밀도
5. 버전 안정성 > 이전 저장값 재사용

이 원칙을 기준으로, 시각적으로 완성도 높고 신뢰감 있는 룰 파라미터 튜닝 화면을 구현하라.
